# Contributing to Git Battle

First off, thank you for considering contributing to Git Battle! We are building an epic, gamified RPG-style battle experience for GitHub profiles, and community contributions are a huge part of making that a reality. You can check out the live project here: [https://git-battle-gamma.vercel.app/](https://git-battle-gamma.vercel.app/).

To keep development organized and ensure no one wastes their time working on overlapping features, **we have a strict "Issue First" policy.** Please read the guidelines below before you start coding.

## 📜 The Golden Rule: Issues First, PRs Second

**Do not submit a Pull Request without an assigned, approved Issue.** Whether you want to add custom player avatars, design new class-specific attack animations, balance the boss fight mechanics, or fix a bug with the health bar scaling, it must start as an Issue. Pull Requests that appear out of nowhere without an attached Issue will be closed.

## 🛠️ How to Contribute (Step-by-Step)

### 1. Find or Create an Issue
Before writing any code, check our [Issues](#) tab to see if your idea or bug is already being discussed.
* **If it exists:** Jump to Step 2.
* **If it doesn't exist:** Open a new Issue. Describe the bug you found or the feature you want to build in detail.

### 2. Ask to be Assigned
Once you find an Issue you want to tackle (or after you've created one), leave a comment asking to work on it:
> *"Hey, I'd love to take this on! Can you assign this to me?"*

**Wait for a maintainer to assign the Issue to you.** Once your profile picture shows up under the "Assignees" section on the right sidebar, you are clear to start coding!

### 3. Fork and Branch
1. Fork the Git Battle repository to your own GitHub account.
2. Clone your fork locally.
3. Create a new branch for your feature or bug fix. 
   * *Pro-tip: Use descriptive branch names like `feature/new-attack-animations` or `bugfix/health-bar-scaling`.*

### 4. Write Your Code
Work your magic! As you build, please keep your code clean and try to match the existing style of the repository. 

### 5. Submit a Pull Request (PR)
Once your code is ready and tested:
1. Push your branch to your forked repository.
2. Open a Pull Request against our `main` branch.
3. **Crucial Step:** In your PR description, you *must* link the Issue you were assigned to by typing `Closes #IssueNumber` (e.g., `Closes #42`). 

### 6. Review and Merge
A maintainer will review your code. We might request a few changes or ask some questions. Once everything looks good, we will approve and merge your PR into the main game!

---

### Need Help?
If you're stuck on where to start, look for Issues tagged with `good first issue` or `help wanted`. We're excited to see what you bring to the battlefield!
